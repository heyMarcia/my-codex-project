# 跨境电商 RPA 应用资产诊断

版本：V0.4

这份文档是 `SKILL.md` 的中文对照版，给人阅读和维护使用。`SKILL.md` 保持纯英文，方便 AI 执行；本文件保持纯中文，方便业务侧检查。

## 核心定位

本技能用于根据客户现有 RPA 应用资产，诊断客户当前自动化使用画像，并推荐相邻的 AI+RPA 拓展机会。它不是“自动化场景大全推荐器”，而是从客户已经做过什么出发，反推出下一步最值得验证的需求入口。

## 执行契约

必须从客户已有 RPA 应用资产出发。

1. 先分析客户已经自动化了什么，再判断下一步适合拓展什么。
2. 必须区分已确认事实、合理推断和待验证假设。
3. 优先推荐相邻拓展：同应用下一步、同岗位加深、同数据链路延伸、AI+RPA 升级。
4. 每个 P0/P1 建议都要检查 ERP、API、自研系统、页面稳定性、AI 信任、数据口径和业务责任边界风险。
5. 输出必须服务 CSM 下一步动作：找谁聊、先聊什么、怎么追问、试点选什么。
6. 不要直接生成完整项目方案、报价或实施排期，除非用户在诊断后明确要求。
7. 不要复读机会地图。机会地图只是参照系，不是复制源。

## 适用场景

当用户提供以下任意信息时，使用本技能：

- 客户已有 RPA 应用名称、机器人名称、自动化清单、截图或导出的应用数据。
- 想判断当前 RPA 覆盖了哪些岗位、业务环节或部门。
- 想根据已有应用判断下一步适合拓展什么需求。
- 想对照跨境电商自动化机会地图，判断覆盖和缺口。
- 想判断已有 RPA 应用是否适合升级成 AI+RPA 试点。
- 用于续费、增购、AI 试点、高层汇报、业务拜访或需求挖掘前的客户资产诊断。

不适用场景：

- 客户尚未落地任何 RPA 应用，或没有应用清单：应使用客户初判类技能。
- 用户只有客户行业、类目、平台、团队规模，没有应用名称：不要假装做资产诊断。
- 用户要求排查单个技术故障：应走技术排障。
- 用户要求评估稳定性、成功率、运行频次：需要运行数据，应用名称只能支持名称级诊断。

## 默认参考资料

默认机会地图文件：

```text
references/跨境电商：自动化机会地图.xlsx
```

使用本技能时，应先尝试打开该文件，并将其作为场景匹配、覆盖分析和查漏补缺的主要参照。如果用户另行提供新版机会地图，优先使用用户提供的文件。

读取 Excel 时，优先使用 `openpyxl.load_workbook(..., read_only=False, data_only=True)`。部分工作表在只读模式下可能因为维度标记异常只能读到 A1。

## 输入处理

最小输入：

```text
客户名称：可选
RPA 应用名称列表：
1.
2.
3.
```

增强输入包括：客户类型、主营平台、类目、具体品类、供货模式、当前 ERP/系统、团队规模、SKU 数、店铺数、应用使用情况和当前目标。

字段缺失时，不要停止，也不要编造事实。先输出可判断内容，再把缺失信息放进“待确认问题”。

判断边界：

- 只有应用名称时，只能做保守归类、岗位/工作流推断、当前使用画像、相邻拓展方向和待确认问题。
- 不要在证据不足时断言客户类型、组织结构、老板意图、真实痛点或 ROI。
- 缺少 ERP/系统信息时，要标注 ERP 替代风险未确认。
- 缺少运行数据时，不要把应用数量等同于业务价值。

## 必须遵循的分析流程

按以下步骤执行：

1. 判断输入完整度和当前任务目标。
2. 标准化应用名称，同时在输出中保留原始应用名称。
3. 拆解每个应用名称中的平台、业务对象、操作动作、数据/文件类型和业务意图。
4. 按“应用名称 -> 业务对象 -> 操作动作 -> 岗位 -> 工作流 -> 机会点”匹配。
5. 判断置信度、自动化层级和痛点类型。
6. 聚合客户当前 RPA 使用画像。
7. 对照机会地图做岗位/工作流覆盖分析。
8. 只查找与已有应用或数据链路相关的缺口。
9. 生成相邻链路拓展建议。
10. 识别 AI+RPA 升级路径。
11. 做风险过滤和 P0/P1/P2 优先级排序。
12. 输出 CSM 下一步作战建议。
13. 执行交付前自检。

## 应用归类规则

### 岗位匹配

优先看业务对象，其次看操作动作，再看平台信息，客户背景只作为修正项。

| 应用信号 | 优先归属岗位 |
|---|---|
| 广告、ACOS、CPC、SP/SB/SD、Bulk、关键词、否词、预算、Campaign | 广告投放 |
| 账单、结算、对账、发票、税单、费用、利润、回款、凭证 | 财务对账 |
| 订单、面单、发货、物流、仓库、库存、FBA 货件、海外仓 | 仓储物流 |
| 供应商、采购、1688、询价、催单、交期、工厂、补货 | 供应链采购 |
| 评论、邮件、工单、退款、退货、索赔、账号健康、差评 | 客服售后 |
| Listing、标题、五点、变体、刊登、A+、图片 | 运营 |
| BSR、竞品、新品、类目、市场调研、价格带、销量估算 | 产品选品 |
| 达人、KOL、TikTok 达人、社媒、内容、视频 | 市场营销 |
| 日报、月报、看板、绩效、利润归因、管理报表 | 管理决策 |

### 易混词处理

- “库存”不默认归仓储物流。库存同步/更新归仓储物流；补货风险归供应链/仓储；库存金额和周转归管理/财务；FBA 库存赔偿归财务/售后。
- “关键词”不默认归广告。广告关键词和否词归广告；ABA/自然词/搜索排名归运营；竞品/类目关键词归产品选品；Listing 关键词归运营/广告。
- “评论”不默认归客服。竞品评论归产品选品；买家评论回复归客服售后；社媒评论归市场营销；评论痛点提炼归产品/运营。
- “FBA”看后缀。货件创建归仓储物流；库存差异/赔偿归财务/售后；FBA 费用归财务；FBA 补货归供应链/仓储。
- “报表”必须看内容。销售日报、广告日报、财务月报、老板看板、多平台运营月报分别归不同模块。

## 自动化层级

| 层级 | 典型信号 | 当前价值 | 下一步方向 |
|---|---|---|---|
| L1 采集搬运型 | 下载、导出、采集、抓取、上传、同步、录入、截图、归档 | 减少重复操作 | 汇总、清洗、分析、预警 |
| L2 规则处理型 | 汇总、清洗、比对、校验、分类、命名、匹配、转换、拆分、合并 | 进入业务规则层 | 异常识别、原因归因、自动报告 |
| L3 分析预警型 | 分析、监控、预警、评分、归因、诊断、复盘、趋势、异常 | 支撑管理和决策 | AI+RPA 试点、管理层汇报 |
| L4 决策闭环型 | 推荐、策略、自动处理、自动提交、智能分配、动态定价、补货建议、否词建议 | 价值高但风险高 | 人工复核、权限边界、日志留痕 |

## 置信度

- 高：业务对象和操作动作清晰，能稳定匹配岗位/工作流。
- 中：大方向清楚，但可能归属多个岗位。
- 低：名称过于抽象，如“数据处理”“运营表格”“自动化流程”“系统同步”“测试应用”。低置信度应用不能作为强建议依据。

## 覆盖分析和查漏补缺

覆盖分析按岗位/工作流判断，不要求逐条匹配机会地图。

- 高覆盖：已有多个相关应用，说明可能有负责人、有认知、有使用基础。
- 中覆盖：已有少量相关应用，可以继续试探。
- 低覆盖：几乎没有相关应用，不建议贸然强推。
- 未知：应用名称信息不足。

查漏补缺必须受约束：

1. 先看高覆盖岗位下的缺口。
2. 再看与已有应用相邻的数据链路缺口。
3. 再看与客户类型、类目、平台或当前目标强相关的缺口。
4. 最后才考虑高层管理或跨部门协同缺口。

每个缺口必须说明：所属岗位/工作流、为什么是缺口、和已有应用的关系、为什么现在值得聊、优先级和待确认信息。

## 推荐优先级

P0：马上值得聊。通常与已有应用强相关，数据基础已经存在，触达对象明确，价值容易解释，技术风险可控，并且能自然从“当前应用跑完之后下一步怎么办”问出来。

P1：适合试点。价值明显，但需要更多样本、业务规则、人工复核或跨部门协同。

P2：后续储备。当前资产基础不足，需要新部门介入，依赖更多数据打通，容易与 ERP/BI/自研系统重合，或不适合第一次拜访强推。

推荐路径优先级：

1. 同应用下一步。
2. 同岗位加深。
3. 同数据链路延伸。
4. RPA 执行层升级为 AI+RPA 分析层。
5. 根据当前业务目标修正。
6. 对照机会地图查找相关缺口。
7. 全新业务方向，通常只放 P2。

## 风险过滤

每个 P0/P1 建议都必须检查：

| 风险 | 检查问题 | 处理方式 |
|---|---|---|
| ERP 替代风险 | 领星、积加、店小秘、马帮等是否已有类似功能 | 若高度覆盖，改讲跨平台、长尾、非标流程补充 |
| API 替代风险 | 是否更适合标准接口同步 | 不包装成 RPA 优势，改讲流程编排或非标补充 |
| 自研替代风险 | 客户是否有 IT/开发能力 | 强调快速验证、低成本试点、业务自助配置 |
| 页面稳定性风险 | 是否有频繁改版、验证码、风控 | 降低承诺，优先文件/API，增加人工兜底 |
| AI 信任风险 | 是否涉及广告、财务、定价、侵权、退款等高风险判断 | 只做候选清单和人工确认，不做全自动决策 |
| 业务责任风险 | AI 建议错了谁负责 | 增加审批、权限边界和日志 |
| 数据口径风险 | 多平台/多账号指标是否一致 | 先用样本验证，再承诺全量 |
| 组织协同风险 | 是否需要跨部门规则、权限、数据归属 | 先做单部门小试点 |

不要包装成“AI 自动否词”“AI 自动调价”“AI 自动定价”。应包装成“AI 生成候选清单，业务人工确认，RPA 批量执行”。

## 输出密度

默认控制：

- 应用画像：一段短文。
- 应用归类表：少于 20 个应用完整列出；超过 20 个时可聚合并说明规则。
- 机会地图覆盖：只保留高/中覆盖和必要的低覆盖。
- 查漏补缺：最多 5 个。
- P0 建议：最多 3 个。
- P1 建议：最多 3 个。
- P2 建议：最多 2 个。
- AI+RPA 路径：最多 5 条。
- 待确认问题：3-5 个。
- 每个主要模块后都要有简短模块小结。

## CSM 追问话术

优先问和已有应用强相关的问题：

- 这个应用现在是谁在用？每天/每周跑几次？
- 跑完之后结果给谁看？会进入日报、周报或经营决策吗？
- 这个应用解决的是节省时间，还是影响业务判断？
- 目前最有价值的 3 个应用是哪几个？为什么？
- 哪些应用已经低频、停用，或被 ERP/自研系统替代？
- 数据下载下来后，下一步是谁分析？有没有固定模板或规则？
- 哪些异常现在仍然靠人发现？
- 如果数据已经能稳定拿到，下一步希望 AI 帮忙看什么？
- 哪些判断必须由业务人工确认，不能全自动？
- 这个场景更适合提醒、建议，还是自动执行？

## 交付前自检

1. 是否基于客户已有应用，而不是泛泛推荐？
2. 是否说明信息完整度与判断边界？
3. 是否区分已确认事实、合理推断、待验证假设？
4. 每个应用或应用组是否有岗位、工作流、层级、置信度和依据？
5. 每个 P0/P1 是否能说清来源依据？
6. 是否覆盖 ERP/API/自研/页面稳定性/AI 信任/业务责任风险？
7. 查漏补缺是否只列相关缺口？
8. 是否避免复读机会地图？
9. CSM 下一步问题是否具体可用？
10. 是否避免把高风险 AI 判断包装成无复核的全自动决策？
11. 每个主要模块是否有模块小结？
12. 输出是否少而准？
